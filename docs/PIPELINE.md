# Semantic Particle Filter Pipeline

This document explains the working principle of the repository at a level that matches the code and the paper.

## Inputs

The main pipeline in `scripts/spf_lidar.py` consumes:

- RGB images for semantic instance segmentation
- depth images for local metric projection
- LiDAR scans for geometric observation support
- GNSS positions for global anchoring
- a GeoJSON vineyard map containing stable row landmarks

The current implementation focuses on two landmark classes:

- support poles
- vine trunks

## 1. Semantic detection

Each RGB frame is processed by the trained YOLO segmentation model in `models/yolo.pt`. The output masks isolate stable vineyard structures rather than transient appearance cues.

## 2. Local BEV projection

Depth is used to project the detected landmarks into a robot-centric bird's-eye view. The resulting BEV landmarks act as semantic observation regions near the robot.

## 3. Semantic labelling of LiDAR returns

The BEV landmark regions are used to label nearby LiDAR beams by semantic class. This converts the raw LiDAR scan into class-aware observations that can be compared against the map.

## 4. Semantic wall map

The GeoJSON map is converted into row-wise landmark groups and piecewise-linear segments. These segments act as semantic walls that provide stronger row-level structure than isolated point landmarks.

## 5. Particle filter update

The filter maintains multiple pose hypotheses and updates them using:

- odometry-driven motion propagation
- semantic wall agreement
- background free-space consistency
- GNSS agreement
- corridor regularisation relative to nearby row segments

This combination is what makes the method more robust in repetitive vineyard geometry, headlands, and row transitions.

## 6. Dynamic GNSS weighting

GNSS is not treated as a fixed-strength correction. The implementation adapts its influence based on how much semantic evidence is available in the current frame, so GNSS matters more when semantic support is weak.

## 7. Resampling and pose extraction

Particles are reweighted, resampled when necessary, and converted into a final pose estimate. The exported trajectory can optionally be smoothed before being written as a TUM file.

## Default paper configuration

The paper-facing experiments use a fixed configuration for the main SPF++ runs, including:

- frame stride `4`
- `miss_penalty=4.0`
- `wrong_hit_penalty=4.0`
- `gps_weight=0.5`
- `semantic_sigma=0.05`
- `gps_sigma=1.1`
- `corridor_weight=0.30`
- `particle_count=100`

For the runnable paper defaults, use `scripts/spf_lidar.py` together with the experiment protocol documents under `docs/`.

## Outputs

The main outputs are:

- TUM trajectories for evaluation
- per-frame runtime profiles when enabled
- diagnostic plots and overlays
- aggregate metrics generated by the evaluation scripts

Metric definitions are documented in `docs/EXPERIMENT_RUNNER_AND_METRICS.md`.
